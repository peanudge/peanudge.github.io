디자인 시스템을 정의할 수 있는 개발자

December 5, 2022 · 11 min read
Jiho Shon
Frontend Developer in miso.inc

디자인 시스템은 5가지로 정의할 수 있을 것 같습니다.

  1. 디자인 시스템은 일관성을 지켜줘서 사용자의 경험을 개선해준다.
  2. 재사용성을 올려서 일의 생산성을 높여준다.
  3. 위 두가지를 통해 비지니스 임팩트를 줄 수 있다.
  4. 디자인 시스템은 항상 진화한다.
  5. 디자인 시스템은 정의하는 사람마다 정의 자체가 다를 수 있다.

그리고 개인적으로 조사를 하면서 느꼈지만, 개발자 입장에서는 디자인시스템 컴포넌트 라이브러리는 사용자에게 어느정도 노출시킬 것인가가 중요한 것 같다. 배시시(배민의 백오피스 디자인시스템)는 organism 수준의 컴포넌트만 노출시켜서 사용자들의 일관적인 백오피스 개발을 유도하는 것 같았다. (운영의 효율성을 높이기 위해)

너무 위키만 보고 만든 부분이 많음.

다시 정리할 필요성이 있음.

왜필요하고 언제 디자인시스템을 사용해야지? https://story.pxd.co.kr/1434

우형, 배시시 디자인 시스템 철학 및 가이드 https://www.youtube.com/watch?v=yR1g_A1-xFs

이거 두개 활용해보기.

👀 요새 자주 언급되는 디자인 시스템?​

유명 컨퍼런스, 유니콘 기업이 된 스타트업 등등에 "디자인 시스템을 이렇게 도입해서 생산성을 높였다."와 같은 이야기가 나오곤 합니다.

규모가 작거나 디자인 시스템의 필요성(서비스가 하나일 경우)을 못 느끼는 작은 스타트업에 다니는 개발자들은 매번 기능을 개발할 때 디자이너가 준 시안을 처음부터 scratch해서 만들거나 이 방식이 피곤하게 느껴진다면 작게 컴포넌트화를 진행해서 하나의 프로젝트에 한정해서 재사용을 하게됩니다.

이러한 방식과 차이점은 무엇이고 언제 디자인 시스템을 도입해야할까요?

그럼 왜 필요한 시점이 올까? 그리고 디자인 시스템이 구체적으로 뭐라고 정의할 수 있을까?

그리고 컴포넌트 기반 vs 파운데이션 기반 디자인 시스템 중 어떤 방식이 어떤 상황에 적합할까?

✍️ 디자인 시스템에 대한 정의?​

여러 자료, 블로그들을 살펴봤을 때 다양한 표현이 있었지만 거의 맥락을 같았습니다.

  • 디자인 시스템은 디자인 원칙, 규격, 다시 사용할 수 있는 UI 패턴과 컴포넌트, 코드를 포괄하는 종합 세트입니다

  • A Design system is a set of interconnected patterns and shared practices coherently organized

하지만 각 디자인 시스템마다 포괄하는 범위는 달랐습니다. pattern libraries의 역할만하는 경우도 존재하기도 했습니다.

디자인 시스템의 역할​

Design systems aid in digital product design and development of products such as applications or websites

"어플리케이션이나 웹사이트 같은 product의 디자인과 개발에 도움을 준다" 추상적이지만 명확한 표현인 것 같네요.

디자인 시스템의 구성 🔧​

They may contain, but are not limited to, pattern libraries, design languages, style guides, coded components, brand languages, and documentation.

pattern libraries, design language, style guides, coded components, brand languages, and documentation이지만 제한되지는 않습니다. 이러한 참조 자료들은 다른 팀들(디자이너, 개발자, PM를 포함해서)이 Product를 디자인하고 빌드하는데 도움을 주는 역할을 합니다.

디자인 시스템 관련해서 리서치를 해볼 때 위와 같이 다양한 키워드들이 자주 등장하는데 coded components를 제외하곤 개발자 입장에서는 다소 낯선 이름들이 가득합니다.

✅ 해당 키워드들에 대한 정확한 이해가 필요해보입니다.

pattern libraries, design language, style guides, coded components

디자인 시스템의 장점​

https://velog.io/@korean-davinci/%EB%94%94%EC%9E%90%EC%9D%B8-%EC%8B%9C%EC%8A%A4%ED%85%9C%EC%9D%B4%EB%9E%80

  • 재사용성있는 컴포넌트들과 공유되는 논리적인 근거를 통해 빠른 Product 개발을 도와준다.
  • 일관성있는 user expierence 과 design language를 통해 더 나은 Product
    • 공동의 디자인 자산 확충 (팀의 규모가 커질수록 공동 지식을 유지하는 게 어려워집니다.)
  • technical, design debt 감소를 통해 개선된 유지보수와 확장성
    • 커뮤니케이션 기준 마련
  • product 팀이 user needs를 해결하는데 더욱 집중할 수 있습니다.

디자인 시스템 기원​

Mainstream interest about pattern languages for UI design surged again by the opening of Yahoo! Design Pattern Library in 2006[12] with the simultaneous introduction of Yahoo! User Interface Library (YUI Library for short). The simultaneous introduction was meant to allow more systematic design than mere components which the UI library has provided.

야후의 시작과 함께 UI 디자인을 위한 패턴이 인기를 얻게되었던 것 같습니다. 이러한 등장은 단순히 UI 라이브러리가 제공하는 컴포넌트보다 더욱 시스템적으로 구성된 디자인을 가능하게 해줬습니다.

여기서 주목할만한 내용은 more systematic design than mere components which the UI library has provided 으로 볼 수 있습니다. 단순히 개발자가 오픈 소스 UI 라이브러리를 가져와서 일관되게 사용하는 것과 디자인 시스템은 확실히 다름을 말하는 것으로 알려진 것 같고 그 차이를 알아가는 것이 디자인 시스템에 대한 학습이라고 생각이 듭니다.

Technical challenges of large-scale web projects led to the invention of systematic approaches in the 2010s, most notably BEM[13] and Atomic Design. The book about Atomic Design[14] helped popularize the term "Design System" since 2016.

거대한 규모의 웹 프로젝트들을 만드는 어려움들이 시스템적인 접근을 발명으로 이어졌고 Atomic Design 에 대한 책이 Design System의 대중화에 도움이 많이 되었다.

✅ 위 내용을 확인해보니 대규모 웹 프로젝트를 하면서 생기는 디자인 관련 어려움들을 확인해볼 필요성이 있어보입니다.

✅ Atomic Design 에 대한 이해를 하면 디자인 시스템도 자연스레 이해할 수 있어보입니다.

(논외) Difference between pattern languages and design systems and UI kits​

// 이 내용은 빼도 될듯함.

디자인 시스템과 지금까지 우리가 해왔던 것들의 구분을 이해하는데 도움이 많이 되는 내용이었습니다.

디자인 시스템의 기원을 살피다보면 pattern language 라는 용어가 자주 나오는데 디자인 시스템와 동일시되어 사용되는 개념처럼 보였습니다. 하지만 엄연히 다른 개념임을 여기서 알 수 있습니다.

A pattern language allows its patterns to exist in many different shapes and forms – for example, a login form, with an input field for username, password, and buttons to log in, register and retrieve lost password is a pattern, no matter if the buttons are green or purple.

pattern language는 다른 형태와 모양이 존재할 수 있는 패턴 그 자체를 의미하게 됩니다.

이 개념으로 이해한 pattern language는 최근 많이 들었던 headless UI가 pattern language만을 제공하는 것 같았습니다.
